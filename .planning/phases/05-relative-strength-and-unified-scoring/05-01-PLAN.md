<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| YAML config -> RSL evaluator | Configurable thresholds and disabled behaviors influence setup validity. |
| Completed daily bars and return series -> RSL evaluator | Offline fixture data can be incomplete, stale, invalid, or manipulated. |
| SymbolData/regime -> RSL evaluator | Shared readiness and entry-gate state determine whether a setup can be valid. |
| RSL setup output -> later scoring/ranking | Evidence is consumed later but must not carry trade/order intent. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | `config/relative_strength.yaml` | mitigate | Loader must require `paper_trading_only: true`, completed daily-bar timing, and disabled behavior flags before returning config. |
| T-05-02 | Spoofing | `evaluate_relative_strength_leader` input data | mitigate | Use `SymbolData.future_signal_ready(..., stale=...)`, completed-bar validation, and finite numeric checks; missing/invalid/stale data rejects. |
| T-05-03 | Tampering | Relative strength benchmark handling | mitigate | Tests must prove SPY RS20/RS60 are hard gates per D-02/D-03 and weak QQQ is evidence-only per D-02/D-04. |
| T-05-04 | Elevation of Privilege | RSL setup output | mitigate | Static tests must prove no orders, sizing, portfolio mutation, entry/stop/target/order intent, Paper/Live deployment, Telegram delivery, credentials, fake performance, or classifications are emitted. |
| T-05-05 | Repudiation | RSL rejected output | mitigate | Preserve `NumericEvidence` and exact rejection reason explanations for every failed gate. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | This plan installs no packages. If an executor needs a new package, stop and run the package legitimacy gate before any install. |
</threat_model>
