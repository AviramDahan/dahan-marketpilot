<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Environment/secret store -> Telegram config | External token/chat ID availability controls real delivery but must never enter repository files. |
| NotificationDomainEvent -> Telegram payload | Sanitized internal events become outbound message text. |
| Telegram API response -> delivery result | External API success/failure is untrusted and must be parsed into typed non-authoritative outcomes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-12 | Information Disclosure | Telegram config and logs | mitigate | Reject committed secret values and redact token/chat ID/credential-like fields in results, payloads, docs, and tests. |
| T-08-13 | Tampering | Telegram delivery result parser | mitigate | Parse `ok`, `result`, `error_code`, `description`, and retry parameters into typed statuses. |
| T-08-14 | Denial of Service | Telegram rate limits | mitigate | Local conservative rate limiter returns rate_limited and does not use paid broadcast. |
| T-08-15 | Elevation of Privilege | Delivery success | mitigate | Tests prove delivery success or failure cannot approve Paper modes, orders, lifecycle transitions, reconciliation, or protective recovery. |
| T-08-16 | Repudiation | Delivery failures | mitigate | Record typed sanitized delivery results with event type and correlation ID. |
| T-08-SC | Tampering | package installs | accept | No npm, pip, or cargo install tasks exist in this plan; use existing dependencies and Python standard library only. |
</threat_model>
