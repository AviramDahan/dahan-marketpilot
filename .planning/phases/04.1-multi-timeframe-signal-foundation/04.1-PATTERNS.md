# Phase 4.1 Patterns

## Existing Patterns To Reuse

- Use fail-closed YAML config loaders from existing setup/config modules.
- Use immutable dataclass-style domain contracts where existing project code
  already uses them.
- Keep setup evaluators evidence-only and separate from scoring, orders,
  portfolio state, Paper Trading, Telegram, and backtesting.
- Preserve deterministic offline pytest coverage.
- Update docs in the same phase as contract changes.

## New Pattern To Introduce

Timeframe-aware contracts should be shared instead of duplicating Daily/4H/1H
logic inside each setup. The intended direction is:

- strategy mode config is central and typed
- completed-bar metadata is reusable
- timeframe readiness is independent
- setup timing is generalized
- setup modules adapt behavior by contract, not by copy-pasting whole rule sets

Final file names are implementation choices, but expected areas include
`marketpilot/timeframes.py`, `marketpilot/config.py`, and
`marketpilot/setups/base.py` if they fit the existing codebase.
