# Plan 04.1-02: Completed-Bar Models, SetupTiming, And Readiness

## Objective

Introduce reusable timeframe-aware completed-bar contracts and generalize setup
timing so future setup evidence can identify the exact completed signal
timeframe.

## Tasks

1. Add reusable models such as `BarTimeframe`, `CompletedBar`,
   `BarCompletionStatus`, and `BarSessionMetadata` or equivalent names.
2. Generalize `SetupTiming` so it preserves strategy mode, signal timeframe,
   timestamp, bar start/end, completion status, exchange timezone,
   regular-hours status, partial-session status, freshness, source resolution,
   and later-valid-execution requirement.
3. Add independent readiness contracts for Daily, 4H, and 1H.
4. Make mandatory readiness fail closed while keeping optional 1H absence
   evidence-only in the optional-1H mode.
5. Add deterministic tests for no future bars, no incomplete bars, no same-bar
   execution assumptions, early closes, holidays, DST, RTH-only behavior, and
   stale-data rejection.

## Verification

- `python -m pytest tests/test_*timeframe* tests/test_*setup* -q`
- `python -m pytest -q`

## Acceptance Criteria

- Completed-bar metadata and setup timing are timeframe-aware.
- `daily_only` behavior remains compatible with existing completed daily-bar
  setup tests.
- Mandatory timeframe data fails closed and optional 1H cannot override hard
  failures.
